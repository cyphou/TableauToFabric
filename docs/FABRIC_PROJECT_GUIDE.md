# Microsoft Fabric Project Guide

Guide to understanding the **Fabric artifacts** generated by the Tableau-to-Fabric migration tool, including how to work with them locally, deploy them to a Fabric workspace, and manage them in source control.

---

## 1. Artifact Overview

The migration tool generates **six artifact types** that together form a complete analytics solution in Microsoft Fabric:

```
output/
├── <Name>.Lakehouse/        ← Delta table definitions
├── <Name>.Dataflow/         ← Power Query M data ingestion
├── <Name>.Notebook/         ← PySpark ETL & transforms
├── <Name>.SemanticModel/    ← DirectLake TMDL model
├── <Name>.Pipeline/         ← Orchestration workflow
└── <Name>.pbip              ← Power BI Report project
    ├── <Name>.SemanticModel/
    └── <Name>.Report/
```

### Data Flow

```
         ┌──────────────┐
         │  Data Sources │  (Snowflake, SQL Server, CSV, ...)
         └──────┬───────┘
                ↓
    ┌────────────────────────┐
    │   Dataflow Gen2        │  Power Query M → data ingestion
    │   (mashup.pq)          │
    └────────────┬───────────┘
                 ↓
    ┌────────────────────────┐
    │   Lakehouse            │  Delta tables (physical storage)
    │   (DDL scripts)        │  Includes materialised calc columns
    └────────────┬───────────┘
                 ↓
    ┌────────────────────────┐
    │   Notebook             │  PySpark ETL → write to Delta
    │   (etl_pipeline.ipynb) │  Calc column transformations
    └────────────┬───────────┘
                 ↓
    ┌────────────────────────┐
    │   Semantic Model       │  DirectLake TMDL → reads from Lakehouse
    │   (model.tmdl)         │  DAX measures, relationships, RLS
    └────────────┬───────────┘
                 ↓
    ┌────────────────────────┐
    │   Power BI Report      │  PBIR visuals, slicers, pages
    │   (.pbip project)      │  60+ visual types, bookmarks
    └────────────────────────┘

    ┌────────────────────────┐
    │   Pipeline             │  Orchestrates: Dataflow → Notebook → Refresh
    │   (pipeline_definition)│  Retry policies, timeouts, dependencies
    └────────────────────────┘
```

## 2. Lakehouse Artifact

### Structure

```
<Name>.Lakehouse/
├── lakehouse_definition.json    # Lakehouse item metadata
├── table_metadata.json          # Table/column inventory
└── ddl/
    ├── Orders.sql               # CREATE TABLE ... USING DELTA
    ├── Customers.sql
    └── Calendar.sql
```

### DDL Example

```sql
CREATE TABLE IF NOT EXISTS Orders (
    OrderID         BIGINT,
    CustomerID      BIGINT,
    OrderDate       DATE,
    Quantity         BIGINT,
    UnitPrice        DOUBLE,
    Discount         DOUBLE,
    Revenue          DOUBLE,        -- Materialised calc column
    Status_Label     STRING,        -- Materialised calc column
    PRIMARY KEY (OrderID)
) USING DELTA;
```

### Key Points
- All tables use **Delta Lake** format
- Calculated columns appear as **physical columns** (not DAX expressions)
- The DDL files are reference scripts — in Fabric, tables are created by the Dataflow/Notebook pipeline

## 3. Dataflow Gen2 Artifact

### Structure

```
<Name>.Dataflow/
├── dataflow_definition.json     # Dataflow item metadata + Lakehouse destination
├── mashup.pq                    # Combined Power Query M document
└── queries/
    ├── Orders.m                 # Individual M query per table
    ├── Customers.m
    └── Calendar.m
```

### M Query Example

```
let
    Source = Snowflake.Databases("account.snowflakecomputing.com", "WAREHOUSE"),
    DB = Source{[Name="DATABASE"]}[Data],
    Schema = DB{[Name="PUBLIC"]}[Data],
    Orders = Schema{[Name="ORDERS"]}[Data],
    #"Changed Types" = Table.TransformColumnTypes(Orders, {
        {"OrderID", Int64.Type},
        {"CustomerID", Int64.Type},
        {"OrderDate", type date},
        {"Quantity", Int64.Type},
        {"UnitPrice", type number},
        {"Discount", type number}
    }),
    // Materialised calculated column: Revenue
    #"Added Revenue" = Table.AddColumn(#"Changed Types", "Revenue",
        each [Quantity] * [UnitPrice] * (1 - [Discount]), type number),
    // Materialised calculated column: Status Label
    #"Added Status_Label" = Table.AddColumn(#"Added Revenue", "Status_Label",
        each if [Status] = "Active" then "Active" else "Inactive", type text)
in
    #"Added Status_Label"
```

### Key Points
- Each query loads data **into a Lakehouse Delta table**
- Calculated column logic is injected as `Table.AddColumn()` steps
- The `dataflow_definition.json` configures the Lakehouse destination

## 4. Notebook Artifact

### Structure

```
<Name>.Notebook/
├── etl_pipeline.ipynb           # PySpark ingestion + transforms
└── transformations.ipynb        # Calc column logic (standalone)
```

### PySpark Cell Example

```python
from pyspark.sql import functions as F

# Read source data
df = spark.read.format("delta").load("Tables/Orders")

# Materialised calculated column: Revenue
df = df.withColumn("Revenue",
    F.col("Quantity") * F.col("UnitPrice") * (1 - F.col("Discount")))

# Materialised calculated column: Status Label
df = df.withColumn("Status_Label",
    F.when(F.col("Status") == "Active", "Active").otherwise("Inactive"))

# Write back to Delta
df.write.format("delta").mode("overwrite").saveAsTable("Orders")
```

### Key Points
- Notebooks are valid `.ipynb` files with Fabric metadata
- Include Lakehouse attachment and Spark pool configuration
- Alternative materialisation path for calculated columns

## 5. Semantic Model Artifact (DirectLake TMDL)

### Structure

```
<Name>.SemanticModel/
├── .platform                    # Fabric Git integration manifest
└── definition/
    ├── model.tmdl               # Model-level config (DirectLake)
    ├── database.tmdl            # Database config
    ├── relationships.tmdl       # Table relationships
    ├── expressions.tmdl         # Shared expressions
    ├── roles.tmdl               # RLS role definitions
    └── tables/
        ├── Orders.tmdl          # Columns + measures
        ├── Customers.tmdl
        └── Calendar.tmdl
```

### TMDL Examples

**model.tmdl** — DirectLake mode:
```
model Model
    culture: en-US
    defaultPowerBIDataSourceVersion: powerBI_V3
    sourceQueryCulture: en-US
```

**tables/Orders.tmdl** — Table with DirectLake partition:
```
table Orders

    column OrderID
        dataType: int64
        sourceColumn: OrderID
        summarizeBy: none

    column Revenue
        dataType: double
        sourceColumn: Revenue          // ← physical column, not DAX expression
        summarizeBy: sum

    measure TotalRevenue = SUM('Orders'[Revenue])

    measure AvgOrderValue = DIVIDE(SUM('Orders'[Revenue]), COUNT('Orders'[OrderID]))

    partition Orders = entity
        entityName: Orders
        schemaName: dbo
        expressionSource: DatabaseQuery
```

**relationships.tmdl**:
```
relationship rel_Orders_Customers
    fromColumn: Orders.CustomerID
    toColumn: Customers.CustomerID
    crossFilteringBehavior: oneDirection
```

### Key Points
- **DirectLake** mode: `partition = entity` references Lakehouse Delta tables
- Calculated columns use `sourceColumn` (physical), not `expression` (DAX)
- Measures use standard DAX expressions
- RLS roles use `USERPRINCIPALNAME()` filters

## 6. Pipeline Artifact

### Structure

```
<Name>.Pipeline/
├── .platform                    # Fabric Git integration manifest
├── pipeline_definition.json     # Pipeline activities and dependencies
└── pipeline_metadata.json       # Migration metadata
```

### Pipeline Stages

```
Stage 1: Dataflow Refresh      → Ingest data from sources into Lakehouse
    ↓ (on success)
Stage 2: Notebook Execution    → Run PySpark ETL, materialise calc columns
    ↓ (on success)
Stage 3: Semantic Model Refresh → DirectLake picks up fresh Delta data
```

### Key Points
- Activities have **dependency chains** (sequential execution)
- Configurable **retry policies** and **timeout** values
- Can be triggered manually or on a schedule in Fabric

## 7. Power BI Report Artifact

### Structure

```
<Name>.pbip
├── <Name>.pbip                               # Project file (double-click to open)
├── <Name>.SemanticModel/
│   ├── definition.pbism
│   ├── .platform
│   └── definition/
│       ├── model.tmdl                        # DirectLake model
│       ├── database.tmdl
│       ├── relationships.tmdl
│       └── tables/
│           ├── Orders.tmdl
│           └── ...
└── <Name>.Report/
    ├── definition.pbir
    ├── .platform
    └── definition/
        ├── report.json
        ├── version.json
        ├── RegisteredResources/
        │   └── TableauMigrationTheme.json    # Custom theme
        └── pages/
            ├── pages.json
            └── ReportSection/
                ├── page.json
                └── visuals/
                    └── [id]/visual.json
```

### Opening the Project
1. **Power BI Desktop** (December 2025+): double-click the `.pbip` file
2. **Fabric workspace**: import via Git integration or upload
3. **VS Code**: edit TMDL files directly with IntelliSense (install `TMDL` extension)

### Key Points
- Uses **PBIR v4.0** format (definition-based, not Base64)
- 60+ visual type mappings from Tableau marks
- DirectLake semantic model (same TMDL as standalone)
- Custom colour theme from Tableau workbook colours

## 8. Working with TMDL

### Editing in VS Code

Install the **TMDL** extension for syntax highlighting and IntelliSense.

Common modifications:

**Add a new measure:**
```
measure NewMeasure = CALCULATE(SUM('Table'[Column]), FILTER(...))
    formatString: #,##0.00
```

**Add a new relationship:**
```
relationship rel_Table1_Table2
    fromColumn: Table1.ForeignKey
    toColumn: Table2.PrimaryKey
    crossFilteringBehavior: oneDirection
```

**Modify a column:**
```
column MyColumn
    dataType: string
    sourceColumn: MyColumn
    isHidden
    description: "This column is hidden from end users"
```

## 9. Git Integration

Fabric supports **Git integration** with Azure DevOps or GitHub. The `.platform` files in each artifact enable this.

### Recommended Workflow

```bash
# Initialise repository
git init
git add .
git commit -m "Initial Tableau-to-Fabric migration"

# Create branch for manual adjustments
git checkout -b manual-fixes

# Edit TMDL files, query files, notebooks...
# ...

# Commit and merge
git add .
git commit -m "Post-migration adjustments"
git checkout main
git merge manual-fixes
```

### What to Track in Git

| Include | Exclude |
|---------|---------|
| All `.tmdl` files | `__pycache__/` |
| All `.json` metadata | `.env` files |
| `.m` query files | `artifacts/` (generated output) |
| `.ipynb` notebooks | |
| `.sql` DDL scripts | |
| `.platform` manifests | |

## 10. Deployment to Fabric Workspace

### Prerequisites
- Azure AD application with **Fabric API permissions**
- Service Principal or Managed Identity
- Target Fabric workspace ID

### Environment Variables

```bash
export FABRIC_TENANT_ID="your-tenant-guid"
export FABRIC_CLIENT_ID="your-app-client-id"
export FABRIC_CLIENT_SECRET="your-app-secret"
export FABRIC_WORKSPACE_ID="your-workspace-guid"
export FABRIC_LAKEHOUSE_NAME="MyLakehouse"
```

### Deploy via PowerShell (recommended)

The `scripts/` directory contains production-ready PowerShell deployment scripts:

```powershell
# Create workspace
$ws = .\scripts\New-MigrationWorkspace.ps1 -WorkspaceName "My Migration" -CapacityId $capId

# Deploy all artifacts
.\scripts\Deploy-TableauMigration.ps1 -WorkspaceId $ws.WorkspaceId -ProjectDir ./output/MyReport

# Validate
.\scripts\Validate-Deployment.ps1 -WorkspaceId $ws.WorkspaceId -ProjectDir ./output/MyReport
```

Key features:
- **Idempotent** — re-running reuses existing items (no duplicates)
- **429 retry** — automatic backoff on rate limits
- **LRO polling** — handles long-running operations (202 Accepted)
- **Pipeline item references** — automatically rewrites activity references with deployed item IDs
- **Report definition** — includes definition parts at creation time
- **Notebook format** — converts to Fabric `.py` format with required prologue

### Deploy via Python

```python
from fabric_import.deployer import FabricDeployer

deployer = FabricDeployer()

# Deploy individual artifacts
deployer.deploy_lakehouse(workspace_id, 'MyLakehouse', config)
deployer.deploy_dataflow(workspace_id, 'MyDataflow', config)
deployer.deploy_notebook(workspace_id, 'MyNotebook', config)
deployer.deploy_semantic_model(workspace_id, 'MyModel', config)
deployer.deploy_pipeline(workspace_id, 'MyPipeline', config)
deployer.deploy_report(workspace_id, 'MyReport', config)
```

### Deployment Order

Deploy in this order to satisfy dependencies:

1. **Lakehouse** — creates the storage layer
2. **Dataflow Gen2** — depends on Lakehouse for destination
3. **Notebook** — depends on Lakehouse attachment
4. **Semantic Model** — depends on Lakehouse Delta tables
5. **Pipeline** — depends on Dataflow + Notebook + Semantic Model
6. **Power BI Report** — depends on Semantic Model

## 11. Naming Conventions

| Artifact | Convention | Example |
|----------|-----------|---------|
| Lakehouse table | `PascalCase` | `Orders`, `CustomerSegments` |
| Delta column | `PascalCase` | `OrderDate`, `UnitPrice` |
| TMDL measure | `PascalCase` with spaces | `Total Revenue`, `Avg Order Value` |
| TMDL relationship | `rel_From_To` | `rel_Orders_Customers` |
| Dataflow query | Same as table name | `Orders` |
| Notebook cell | Descriptive markdown header | `# Load Orders Table` |
| Pipeline stage | Action-based | `Refresh Dataflow`, `Run Notebook` |

## 12. Troubleshooting

### Common Issues

| Issue | Resolution |
|-------|-----------|
| DirectLake model shows no data | Run the Pipeline to ingest data first |
| `.pbip` won't open in PBI Desktop | Ensure December 2025+ version; enable `.pbip` preview |
| TMDL syntax error | Check indentation (TMDL is whitespace-sensitive) |
| Relationship not recognized | Verify `fromColumn` / `toColumn` match exact column names |
| Calculated column missing values | Check Dataflow M step or Notebook PySpark logic |
| Pipeline stage fails | Check Lakehouse exists, Dataflow/Notebook references are correct |
| Deployment auth error | Verify Service Principal has Fabric API permissions |
| Delta table schema mismatch | Re-run DDL or update column types in Lakehouse |
